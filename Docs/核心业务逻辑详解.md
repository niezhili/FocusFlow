# FocusFlow 核心业务逻辑详解

---

## 一、计时器双路径安全保存机制

### 1.1 问题背景

计时器是 App 的心脏。用户点击"开始"后进入专注状态，如果在计时过程中发生以下异常，必须保证专注数据不丢失：

- **App 进程被系统杀死**（内存不足）
- **用户划掉 App 任务卡片**
- **Service 被系统回收**
- **设备重启**

### 1.2 架构设计：两条并行保存路径

```
┌─────────────────────────────────────────────────────────────────┐
│                        正常路径                                    │
│                                                                  │
│  TimerViewModel.stopTimer()                                      │
│  (ui/timer/TimerViewModel.java)                                  │
│       │                                                          │
│       └──→ FocusRepository.completeSession(sessionId)            │
│            (data/repository/FocusRepository.java)                │
│                │                                                 │
│                └──→ andThen {                                     │
│                      updateSession()  → 写入 actualDuration       │
│                      addTaskDuration() → 累加 task.totalFocusSec  │
│                    }                                             │
│                                                                  │
├─────────────────────────────────────────────────────────────────┤
│                        安全网路径                                  │
│                                                                  │
│  TimerService.onDestroy()                                        │
│  (service/TimerService.java)                                     │
│       │                                                          │
│       └──→ if (isRunning) {                                      │
│                FocusRepository.completeSessionByTaskAndStartTime( │
│                (data/repository/FocusRepository.java)            │
│                    taskId, startTime, elapsedSeconds)             │
│            }                                                     │
│                                                                  │
│  SQL 安全网（data/dao/FocusSessionDao.java）：                    │
│  UPDATE focus_sessions                                           │
│  SET actual_duration = :duration,                                │
│      end_time = :endTime,                                        │
│      is_completed = 1                                            │
│  WHERE task_id = :taskId                                         │
│    AND start_time = :startTime                                   │
│    AND is_completed = 0  ← 关键：只更新未完成的记录               │
└─────────────────────────────────────────────────────────────────┘
```

### 1.3 为什么需要两条路径？

| 场景 | 正常路径 | 安全网路径 |
|------|---------|-----------|
| 用户正常点击停止 | ✅ 使用 | ❌ 不触发 |
| UI 进程被杀 | ❌ 无法执行 | ✅ Service.onDestroy 触发 |
| 用户划掉 App | ❌ 已销毁 | ✅ Service.onDestroy 触发 |

**竞态条件处理**：如果用户正常停止时，Service 的 onDestroy 也同时触发，两条路径会竞争写入。通过 `WHERE is_completed = 0` 条件，先完成的路径会设置 `is_completed = 1`，后完成的路径则匹配不到记录（影响行数为 0），不会重复写入。

### 1.4 关键代码路径

```
app/src/.../focusflow/ui/timer/TimerViewModel.java:startTimer()
  ├── 创建 FocusSession 记录（仅写入 taskId + startTime，is_completed = 0）
  ├── 启动 Observable.interval(1s) 驱动的定时器
  ├── 启动 TimerService（Foreground Service）
  └── 状态切换为 RUNNING

app/src/.../focusflow/ui/timer/TimerViewModel.java:stopTimer()
  ├── 停止 Observable.interval
  ├── executeCompleteSession(sessionId) → 正常路径
  ├── taskRepository.addTaskDuration() → 累加时长
  ├── 停止 TimerService
  └── 状态切换为 STOPPED

app/src/.../focusflow/service/TimerService.java:onDestroy()
  ├── 检查 isRunning 标志
  ├── if (isRunning) → 安全网路径
  │     focusRepository.completeSessionByTaskAndStartTime(taskId, startTime, elapsed)
  └── 释放 WakeLock
```

## 二、三态状态机

### 2.1 状态定义

```
IDLE ──────点击开始──────→ RUNNING ──点击停止──→ STOPPED
  ↑                                                    │
  └───────────────────────自动返回──────────────────────┘

状态颜色：IDLE=白  RUNNING=绿  STOPPED=白
```

### 2.2 为什么只有三个状态？

Tomato Timer 类 App 通常有四种状态（IDLE/RUNNING/PAUSED/STOPPED），但 FocusFlow 特意去掉了 PAUSED 状态：

- **简化 UI**：用户不需要理解"暂停"和"暂停后继续"的区别
- **减少认知负担**：专注就是专注，要么开始要么结束
- **数据可靠性**：避免暂停状态的数据丢失问题

### 2.3 状态转换代码

```java
// app/src/.../focusflow/ui/timer/TimerViewModel.java
public void toggleTimer() {
    switch (currentState) {
        case IDLE:
            startTimer();   // → RUNNING
            break;
        case RUNNING:
            stopTimer();    // → STOPPED
            break;
        case STOPPED:
            // 自动回到 IDLE，用户可开始新计时
            currentState = State.IDLE;
            break;
    }
}
```

## 三、offsetSeconds 偏移量设计

### 3.1 问题

用户可能在进入 App 之前就在计时（通过通知栏），此时恢复计时页时，需要显示正确的已用时间。

### 3.2 方案

```java
// app/src/.../focusflow/ui/timer/TimerViewModel.java
private long offsetSeconds = 0;  // 本次计时开始前已累积的秒数

// 开始新计时：offsetSeconds = 0
// 恢复未完成计时：offsetSeconds = 已记录的实际时长
// 显示时间 = offsetSeconds + 本次计时秒数

public void startTimer() {
    offsetSeconds = 0;  // 新计时
    Observable.interval(1, TimeUnit.SECONDS)
        .subscribe(tick -> {
            // 格式化显示：offsetSeconds + tick
            long totalSeconds = offsetSeconds + tick;
            timerText.postValue(TimeFormatter.formatSeconds(totalSeconds));
        });
}

public void resumeSession(long taskId, String taskTitle) {
    // 从数据库恢复未完成计时
    FocusSession session = focusRepository.getUncompletedSession();
    this.taskId = taskId;
    this.sessionId = session.getId();
    this.offsetSeconds = session.getActualDuration();  // 累加已有时长
    this.startTime = session.getStartTime();
    currentState = State.RUNNING;
}
```

## 四、统计计算引擎

### 4.1 并行加载 + loadingCount 计数器

```java
// app/src/.../focusflow/ui/stats/StatsViewModel.java
public StatsViewModel(FocusRepository focusRepository, TaskRepository taskRepository) {
    this.focusRepository = focusRepository;
    this.taskRepository = taskRepository;

    // 共需要加载 6 项数据
    final int totalLoads = 6;
    AtomicInteger loadingCount = new AtomicInteger(totalLoads);

    // 6 项数据并行加载
    loadTodayStats(loadingCount, totalLoads);
    loadTodaySessions(loadingCount, totalLoads);
    loadTodayCompletedTasks(loadingCount, totalLoads);
    loadConsecutiveDays(loadingCount, totalLoads);
    loadTaskDistribution(loadingCount, totalLoads);
    loadWeekStats(loadingCount, totalLoads);  // 默认周视图
}

private void onLoadComplete(AtomicInteger loadingCount, int totalLoads) {
    // 每完成一项，计数器减 1；当计数器归零时，所有数据加载完毕
    if (loadingCount.decrementAndGet() == 0) {
        isLoading.postValue(false);  // 隐藏加载动画
    }
}
```

**为什么不用 RxJava 的 zip 操作符？** zip 需要所有 Observable 都返回结果才触发，中途任何一个失败都会导致整体失败。而 loadingCount 方案允许部分失败，其他数据正常显示。

### 4.2 连续天数算法

```java
/**
 * 计算连续打卡天数。
 * 从今天开始往前推，统计连续有专注记录的天数。
 *
 * 算法：
 * 1. 从数据库查询今天往前 30 天的所有专注记录
 * 2. 从今天开始，每一天检查是否有记录
 * 3. 如果某天有记录，连续天数+1，继续往前
 * 4. 如果某天没有记录，停止计数
 * 5. 如果今天没有记录，从昨天开始算（今天还没开始）
 */
// app/src/.../focusflow/ui/stats/StatsViewModel.java
private void calculateConsecutiveDays() {
    List<DailyStats> stats = focusRepository.getDailyStats(startDate, endDate)
        .blockingGet();

    // 将结果转为 Set<String>，O(1) 查找
    Set<String> activeDays = new HashSet<>();
    for (DailyStats s : stats) {
        activeDays.add(s.getDate());
    }

    // 从今天开始往前推
    Calendar cal = Calendar.getInstance();
    SimpleDateFormat sdf = new SimpleDateFormat("yyyy-MM-dd", Locale.US);
    int consecutive = 0;

    // 如果今天没有记录，跳到昨天（今天可能还没开始计时）
    String today = sdf.format(cal.getTime());
    if (!activeDays.contains(today)) {
        cal.add(Calendar.DAY_OF_YEAR, -1);
    }

    // 往前遍历，直到遇到没有记录的日期
    for (int i = 0; i < 30; i++) {
        String date = sdf.format(cal.getTime());
        if (activeDays.contains(date)) {
            consecutive++;
            cal.add(Calendar.DAY_OF_YEAR, -1);
        } else {
            break;
        }
    }

    consecutiveDays.postValue(consecutive);
}
```

### 4.3 图表数据补齐

用户可能不是每天都有专注记录，但图表需要显示完整的时间轴。

```java
/**
 * 补齐周视图（7 天）中缺失的日期。
 * 例如：本周周一、周三有记录，周二、周四~周日没有记录。
 * 补齐后：周一=值, 周二=0, 周三=值, 周四=0, ...
 */
// app/src/.../focusflow/ui/stats/StatsViewModel.java
private List<DailyStats> fillWeekDays(List<DailyStats> rawData) {
    Map<String, Long> dataMap = new HashMap<>();
    for (DailyStats s : rawData) {
        dataMap.put(s.getDate(), s.getTotal());
    }

    List<DailyStats> filled = new ArrayList<>();
    Calendar cal = Calendar.getInstance();
    cal.set(Calendar.DAY_OF_WEEK, Calendar.MONDAY);  // 跳到本周一

    SimpleDateFormat sdf = new SimpleDateFormat("yyyy-MM-dd", Locale.US);
    for (int i = 0; i < 7; i++) {
        String date = sdf.format(cal.getTime());
        long total = dataMap.getOrDefault(date, 0L);  // 缺失的填 0
        filled.add(new DailyStats(date, total));
        cal.add(Calendar.DAY_OF_YEAR, 1);
    }
    return filled;
}
```

## 五、DataStore 主题实时同步

### 5.1 问题

用户在设置页修改主题模式或主题色后，所有页面（包括已打开的）必须立即生效。

### 5.2 传统方案的问题

- **SharedPreferences + OnSharedPreferenceChangeListener**：传统方案，但 Compose 无法直接使用
- **重新创建 Activity**：可以，但体验差

### 5.3 FocusFlow 方案：DataStore Flow + DisposableEffect

```kotlin
// app/src/.../focusflow/ComposeEntry.kt
@Composable
fun FocusFlowApp() {
    val context = LocalContext.current
    val app = context.applicationContext as FocusFlowApp
    val preferences = app.getUserPreferences()

    // 订阅 DataStore 的主题变化
    val themeMode = preferences.getThemeModeFlow()
        .collectAsState(initial = ThemeMode.FOLLOW_SYSTEM)
    val themeColorIndex = preferences.getThemeColorIndexFlow()
        .collectAsState(initial = 0)

    // 当主题变化时，FocusFlowTheme 自动重组（recomposition）
    FocusFlowTheme(
        themeMode = themeMode.value,
        themeColorIndex = themeColorIndex.value
    ) {
        MainScreen()
    }
}
```

```java
// app/src/.../focusflow/data/preferences/UserPreferences.java
public Flowable<Integer> getThemeModeFlow() {
    return dataStore.data()
        .map(prefs -> prefs.get(KEY_THEME_MODE))
        .distinctUntilChanged();  // 只有值变化时才通知
}
```

**关键设计**：
- `distinctUntilChanged()` 避免重复通知，减少不必要的 Compose 重组
- DataStore 基于 Kotlin Flow，天然支持 Compose 的 `collectAsState()`
- 修改设置 → DataStore 写入 → Flow 发射新值 → Compose 重组 → 所有页面立即生效

## 六、任务列表的响应式更新

### 6.1 问题

任务列表需要实时反映变化：新增任务、完成任务、删除任务、更新任务时长。

### 6.2 方案：Room Flowable + LiveData

```java
// app/src/.../focusflow/ui/task/TaskListViewModel.java
public TaskListViewModel(TaskRepository taskRepository) {
    this.taskRepository = taskRepository;

    // 每次数据库变化，Room 自动推送新列表
    taskRepository.getActiveTasks()
        .subscribeOn(Schedulers.io())
        .observeOn(AndroidSchedulers.mainThread())
        .subscribe(activeTasks::postValue);

    taskRepository.getCompletedTasks()
        .subscribeOn(Schedulers.io())
        .observeOn(AndroidSchedulers.mainThread())
        .subscribe(completedTasks::postValue);
}
```

```java
// app/src/.../focusflow/data/dao/TaskDao.java
@Query("SELECT * FROM tasks WHERE is_completed = 0 ORDER BY created_at DESC")
Flowable<List<Task>> getActiveTasks();
```

**为什么用 Flowable 而不是 LiveData？**

Room 的 DAO 可以直接返回 `LiveData<List<Task>>`，但在这里选择了 `Flowable`：
- 可以在 IO 线程订阅，避免主线程查询
- 可以组合操作符（throttle、debounce 等）
- 最终通过 `postValue` 转到 LiveData，供 Compose UI 观察

### 6.3 活跃/已完成任务拆分

```java
// 活跃任务：is_completed = 0
taskRepository.getActiveTasks()    → activeTasks LiveData

// 已完成任务：is_completed = 1
taskRepository.getCompletedTasks() → completedTasks LiveData
```

**为什么拆分？**
- 用户可以同时看到两个列表，不需要切换 Tab
- 活跃任务在顶部，已完成任务在底部折叠显示
- 任务完成时，Room 自动从 activeTasks 移除并添加到 completedTasks

## 七、跨天会话的统计归属

### 7.1 问题

用户在 23:55 开始计时，00:25 结束。这个 30 分钟的专注记录应该计入哪一天？

### 7.2 方案：按结束时间（end_time）统计

```sql
-- app/src/.../focusflow/data/dao/FocusSessionDao.java
-- 今日总时长：按 end_time 过滤
@Query("SELECT COALESCE(SUM(actual_duration), 0) " +
       "FROM focus_sessions " +
       "WHERE is_completed = 1 " +
       "AND end_time BETWEEN :dayStart AND :dayEnd")
Single<Long> getTodayTotalSeconds(long dayStart, long dayEnd);
```

**设计理由**：

1. **用户体验**：用户感知的是"今天完成了多少专注"，按结束时间统计更符合直觉
2. **数据一致性**：一条记录必须有明确的归属日期，不能拆分
3. **连续天数计算**：按结束时间计算，避免了跨天记录的歧义

```java
switch (action) {
    case ACTION_START:
        // 1. 创建通知渠道
        // 2. 构建通知（含暂停/停止按钮）
        // 3. startForeground()
        // 4. 获取 WakeLock
        // 5. 播放提示音 + 震动
        // 6. 启动 Observable.interval 定时器
        break;

    case ACTION_PAUSE:
        // 1. 停止 Observable.interval
        // 2. 更新通知为"暂停"状态
        break;

    case ACTION_RESUME:
        // 1. 重新启动 Observable.interval
        // 2. 更新通知为"计时中"状态
        break;

    case ACTION_STOP:
        // 1. 停止 Observable.interval
        // 2. 保存数据（安全网路径）
        // 3. 释放 WakeLock
        // 4. stopForeground() + stopSelf()
        break;
}
```

## 九、环形计时器绘制

### 9.1 绘制步骤

```kotlin
// app/src/.../focusflow/ui/timer/components/CircularTimer.kt
Canvas(modifier = Modifier.fillMaxSize()) {
    val strokeWidth = 12.dp.toPx()
    val arcSize = size.minDimension - strokeWidth

    // 第 1 步：绘制背景环（半透明，360° 完整圆）
    drawArc(
        color = Color.White.copy(alpha = 0.2f),
        startAngle = -90f,   // 从 12 点钟方向开始
        sweepAngle = 360f,   // 完整圆
        useCenter = false,
        style = Stroke(width = strokeWidth, cap = StrokeCap.Round),
        topLeft = Offset(strokeWidth / 2, strokeWidth / 2),
        size = Size(arcSize, arcSize)
    )

    // 第 2 步：绘制前景弧（实色，按比例）
    // 公式：sweepAngle = elapsedSeconds % 3600 / 3600 * 360
    // 60 分钟 = 360°，即 1 分钟 = 6°
    val progress = elapsedSeconds % 3600 / 3600f
    val sweepAngle = progress * 360f

    drawArc(
        color = primaryColor,
        startAngle = -90f,
        sweepAngle = sweepAngle,
        useCenter = false,
        style = Stroke(width = strokeWidth, cap = StrokeCap.Round),
        topLeft = Offset(strokeWidth / 2, strokeWidth / 2),
        size = Size(arcSize, arcSize)
    )
}
```

### 9.2 动画策略

```kotlin
// 使用 animateFloatAsState 实现平滑过渡
val animatedProgress by animateFloatAsState(
    targetValue = progress,
    animationSpec = tween(durationMillis = 300)  // 300ms 平滑过渡
)
```

**为什么用 300ms？** 每秒更新一次，300ms 过渡时间短于更新间隔，确保动画不会堆积。

## 十、单例模式与初始化顺序

### 10.1 数据库单例（DCL 双重检查锁定）

```java
// app/src/.../focusflow/data/local/AppDatabase.java
private static volatile AppDatabase INSTANCE;

public static AppDatabase getInstance(Context context) {
    if (INSTANCE == null) {
        synchronized (AppDatabase.class) {
            if (INSTANCE == null) {
                INSTANCE = Room.databaseBuilder(
                    context.getApplicationContext(),
                    AppDatabase.class,
                    "focusflow.db"
                ).build();
            }
        }
    }
    return INSTANCE;
}
```

**为什么用 `volatile`？** 防止指令重排序。`INSTANCE = new AppDatabase()` 不是原子操作，JVM 可能先分配内存再初始化，另一个线程可能读到未完全初始化的对象。`volatile` 保证可见性和有序性。

### 10.2 初始化顺序

```java
// app/src/.../focusflow/FocusFlowApp.java
@Override
public void onCreate() {
    super.onCreate();
    // 1. 初始化数据库（其他组件依赖数据库）
    AppDatabase.getInstance(this);

    // 2. 创建通知渠道（必须在显示通知之前）
    NotificationHelper.createChannel(this);

    // 3. 初始化 DataStore（非阻塞，异步初始化）
    userPreferences = new UserPreferences(this);
}
// 顺序不能颠倒：数据库必须在 Repository 使用之前就绪
```

## 十一、数据流全链路

```
用户操作
  ↓
Compose UI (app/src/.../focusflow/ui/*/***Screen.kt)
  ↓ 观察 LiveData / collectAsState
ViewModel (app/src/.../focusflow/ui/*/*ViewModel.java)
  ↓ 调用 Repository 方法
Repository (app/src/.../focusflow/data/repository/*Repository.java)
  ↓ 调用 DAO 方法
Room DAO (app/src/.../focusflow/data/dao/*Dao.java)
  ↓ Room 自动生成 SQL
SQLite 数据库 (app/src/.../focusflow/data/local/AppDatabase.java → focusflow.db)
  ↑ Flowable 自动推送数据变更
Repository
  ↑
ViewModel (更新 LiveData)
  ↑
Compose UI (自动重组 recomposition)
  ↑
用户看到更新后的界面
```

**关键点**：Room 的 `Flowable<List<T>>` 是响应式的，任何 INSERT/UPDATE/DELETE 操作都会自动触发下游订阅者更新。这使得 ViewModel 不需要手动调用 "refresh" 方法。

## 十二、技术选型理由

| 决策 | 选择 | 理由 |
|------|------|------|
| 计时器驱动 | RxJava Observable.interval | 与 ViewModel 的 LiveData 天然集成，RxJava 链式调用简洁 |
| 数据库 | Room | Google 官方推荐，编译时 SQL 校验，支持 Flowable/LiveData |
| 偏好存储 | DataStore | 异步、协程友好、类型安全，替代 SharedPreferences |
| UI 框架 | Jetpack Compose | 声明式 UI，自动重组，代码量少 |
| 导航 | Navigation Compose | 类型安全，支持 saveState/restoreState |
| 图表 | Canvas 手绘 | 依赖少，灵活性高，包体积小（无 Vico 等第三方） |
| 前台服务 | Service + WakeLock | 保证计时不被系统杀死，通知栏常驻 |
| 架构 | MVVM | 关注点分离，ViewModel 不持有 Context（除 Application） |
| 主线程安全 | RxJava subscribeOn.io + observeOn.mainThread | 数据库操作在后台线程，UI 更新在主线程 |
