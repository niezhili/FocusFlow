# FocusFlow 答辩问答准备

## 一、架构与设计模式

### Q1. 你的 ViewModel 和 Repository 都用了 Java，而 UI 层用了 Kotlin Compose。为什么选择这种混合语言方案？用 Kotlin 重写 ViewModel 层会有哪些好处？

**答：**

选择混合语言有两方面原因：
1. **历史原因**：项目初期数据层（Entity / DAO / Repository）是先用 Java 搭建的，因为 Java 的类型系统更传统，Room 注解处理器对 Java 的支持也更成熟、文档更丰富。后来 UI 层切换到 Compose，而 Compose 的 API 完全是 Kotlin-first 的（如 `@Composable` 注解、`remember`、`LaunchedEffect` 等），用 Kotlin 写 UI 更简洁自然。
2. **教学导向**：数据层的 Java 代码有大量中文注释，便于课程设计的同学理解 Room / RxJava 的工作原理，Java 的显式类型声明也更适合学习。

如果用 Kotlin 重写 ViewModel 层，好处包括：
- 可以用 Kotlin 协程 + Flow 替代 RxJava，减少第三方依赖，代码更简洁；
- 消除 `LiveData` 中间层（Room 直接返回 `Flow`，Compose 用 `collectAsState()` 观察），减少样板代码；
- `suspend` 函数替代 `Completable` / `Single`，避免 `CompositeDisposable` 的手动管理。

### Q2. 你用了手动依赖注入（`ViewModelProvider.Factory`），而不是 Hilt/Dagger。手动 DI 在你的项目里带来了什么维护成本？如果项目规模扩大 10 倍，你会怎么选择？

**答：**

当前项目有 4 个 ViewModel，每个 ViewModel 需要一个对应的 Factory 类来传递 Repository 和 Application。手动 DI 的维护成本目前体现在：
- 每新增一个 ViewModel，需要手写一个 `ViewModelProvider.Factory` 实现类；
- Repository 和 DAO 的创建需要手动管理依赖顺序（`AppDatabase` → `TaskDao` → `TaskRepository` → `TaskListViewModel`）；
- 没有生命周期感知的 Scope 管理，所有依赖都是全局单例。

如果项目规模扩大 10 倍，我会选择 Hilt，原因：
- Hilt 自动生成 Factory 代码，减少样板；
- 组件有明确的 Scope（`@Singleton`、`@ViewModelScoped`），可以自动管理生命周期；
- 配合 `@HiltViewModel` 注解，View Model 注入一行代码完成；
- 可测试性更好，Hilt 支持 `@Bind` 接口绑定，方便替换测试替身。

当前项目不做 Hilt 迁移是因为：4 个 ViewModel 对应 4 个 Factory，总代码量不到 100 行，引入 Hilt 需要添加依赖、配置 `@HiltAndroidApp`、声明 Module，反而增加了配置复杂度。

### Q3. 你的三层架构是 `Compose → ViewModel → Repository → DAO`，但 Repo 层基本只是对 DAO 做了 `.subscribeOn(Schedulers.io())` 的包装。你认为 Repo 层在这个项目中是必要的抽象，还是过度设计？

**答：**

我认为在**当前项目规模下** Repository 层是必要的，但不是因为它封装了复杂的业务逻辑，而是因为：

1. **线程调度集中管理**：DAO 本身不负责线程调度，每个 DAO 调用都需要 `.subscribeOn(Schedulers.io())`。如果 ViewModel 直接调 DAO，这段代码会散落在 4 个 ViewModel 中。Repository 将线程调度集中在数据层，ViewModel 不关心数据来源的线程模型。
2. **复合操作封装**：`FocusRepository.completeSession()` 不只是调一个 DAO 方法，而是先 `updateSession()` 再 `andThen(addTaskFocusSeconds())`，这是一个原子化的复合操作，需要 Repository 来编排。
3. **扩展预留**：如果未来需要接入网络同步（比如把专注记录同步到云端），只需在 Repository 层增加一个网络数据源，ViewModel 完全无感知。

所以这不是过度设计，而是合理的分层。真正的过度设计出现在"所有操作都无条件抽象出一个接口"——本项目的 Repository 是具体类，不需要 `interface` + `impl`，这是务实的取舍。

## 二、计时器核心逻辑

### Q4. 你的计时器是三态机（IDLE → RUNNING → STOPPED），没有 PAUSED 状态。文档里说这是"刻意简化 UX"，但用户停止计时后再恢复，你如何保证重新开始的是同一个 `FocusSession` 而不是创建新 session？你处理了哪些边界情况？

**答：**

这是本项目核心设计决策之一。没有 PAUSED 状态确实简化了状态机，但带来了"停止后恢复"的问题。我的处理方式：

**关键代码位置**：`TimerViewModel.java:179-222`（`startTimer()`）

当用户从 STOPPED 状态再次点击时，`startTimer()` 会：
1. 判断 `previousState == TimerState.STOPPED`，如果是，则 `offsetSeconds = elapsedSeconds.getValue()`（保留之前的累计时间）
2. 创建一个**全新的** `FocusSession`（`TimerViewModel.java:207-218`），新的 `sessionStartTime = System.currentTimeMillis()`，新的 `currentSessionId` 等待异步回调
3. 启动新的 `Observable.interval(1s)` 定时器，`elapsed = offset + tick + 1`

所以**每次从 STOPPED → RUNNING 都会创建一个新的 FocusSession**，旧 session 已经在第一次 STOPPED 时完成了。用户看到的累计时间是连续的（因为 offsetSeconds 保留了历史值），但数据库里确实是多条 session 记录。

**边界情况处理**：
- **快速点击**：`toggleTimer()` 通过 `current == TimerState.IDLE/STOPPED` 判断，状态切换后立即生效，不会重复创建；
- **应用被杀后恢复**：`init()` 中调用 `getUnfinishedSession()` 查询 `end_time IS NULL AND is_completed = 0` 的记录，如果有且 taskId 匹配，则 `resumeSession()` 恢复（`TimerViewModel.java:321-346`）；
- **退出页面**：`stopAndSave()` 只在 RUNNING 状态调用 `stopTimer()`，避免误操作；
- **ViewModel 销毁**：`onCleared()` 中检查 RUNNING 状态，自动调用 `stopTimer()` 保存数据。

### Q5. 你设计了"双路径保存"机制（正常路径 + 安全网路径），请解释这个竞态条件是什么场景下发生的？为什么 `currentSessionId` 可能为 -1？

**答：**

**竞态条件场景**：用户在极短时间内完成"开始 → 停止"操作（亚秒级）。

**时间线**：
```
t=0ms:   用户点击"开始" → startTimer() 执行
           ├─ timerState = RUNNING (同步，立即生效)
           ├─ Observable.interval(1s) 启动 (同步)
           ├─ focusRepository.startSession(session) 发起异步请求 (异步！)
           └─ startForegroundService() 启动 (同步)
           
t=100ms: 用户点击"停止" → stopTimer() 执行
           ├─ 此时 currentSessionId 仍为 -1（insertSession 的回调还没执行完）
           └─ 走 else 分支 → 安全网路径
```

**根本原因**：`insertSession()` 是 Room 的异步操作，返回 `Single<Long>`。在 `startTimer()` 中，`subscribe()` 的回调 `sessionId -> currentSessionId = sessionId` 是在 IO 线程写入完成后才执行的。如果用户在回调返回之前就点击了停止，`currentSessionId` 还没被赋值（仍是初始化值 -1）。

**双路径设计**（`TimerViewModel.java:261-298`）：
```
正常路径 (sessionId > 0)：
  completeSession(session) → 通过 ID 精确 UPDATE

安全网路径 (sessionId == -1)：
  completeSessionByTaskAndStartTime(taskId, startTime, duration, endTime)
  → UPDATE ... WHERE task_id=? AND start_time=? AND is_completed=0
```

**安全网的关键保障**：`WHERE is_completed = 0` 确保只更新未完成的记录。即使正常路径和安全网路径都执行了（极端情况），第二次执行时 `is_completed` 已经是 1，UPDATE 影响 0 行，不会重复写入。

### Q6. 你的 `offsetSeconds` 设计——`elapsedSeconds` 显示累计时间，但 `actualDuration` 只记录增量。这样做的好处是什么？如何保证 `totalFocusSeconds` 和 `focus_sessions` 数据一致性？

**答：**

**offsetSeconds 的设计意图**（`TimerViewModel.java:186-190`）：

```
场景：用户对任务"写周报"专注了 30 分钟，停止
      再次进入该任务，elapsedSeconds = 1800（累计值），offsetSeconds = 1800
      用户又专注了 15 分钟
      停止时：elapsed = 2700, actualDuration = 2700 - 1800 = 900s（只记录本次增量）
```

**好处**：
- **用户体验**：UI 显示的是累计专注时间，用户能看到"在这个任务上总共花了多久"，而不是每次重置为 0；
- **数据精确**：`focus_sessions` 表每条记录只存本次的 `actualDuration`，这样统计查询（SUM）的结果是准确的，不会因为 offset 导致重复计算。

**数据一致性保证**（`FocusSessionDao.java:141-156`）：

三条防线：
1. **WHERE is_completed = 0**：`completeSessionByTaskAndStartTime` 只更新未完成的 session。正常路径已经完成后，安全网路径的 UPDATE 影响 0 行。
2. **`addTaskFocusSeconds` 的调用时机**：只在 `completeSession` 的 `andThen` 中调用（`FocusRepository.java`），而不是独立调用。如果 completeSession 失败，addTaskFocusSeconds 也不会执行。
3. **`totalFocusSeconds` 是增量更新**（`UPDATE tasks SET total_focus_seconds = total_focus_seconds + ?`），不是覆盖写入——即使极端情况下重复执行了，也会导致数据翻倍。但因为有第 1 条防线，这种情况不会发生。

**潜在不一致风险**：如果 `addTaskFocusSeconds` 执行成功但 `completeSession` 的 `updateSession` 失败，会导致 `totalFocusSeconds` 加了但 session 没被标记为完成。不过 Room 的 `@Update` 和 `Completable` 在异常时会抛出错误，`andThen` 链会被中断，不会出现这种情况。

### Q7. 计时器用 `Observable.interval(1, TimeUnit.SECONDS)` 驱动，每秒更新一次。如果系统休眠或 CPU 被抢占导致间隔不准，你如何保证计时精度？实际用的是系统时间差还是纯计数器累加？

**答：**

**当前实现**：纯计数器累加（`TimerViewModel.java:196-201`）：

```java
timerDisposable = Observable.interval(1, TimeUnit.SECONDS)
    .observeOn(AndroidSchedulers.mainThread())
    .subscribe(
        tick -> elapsedSeconds.setValue(offset + tick + 1),
        ...
    );
```

`tick` 从 0 开始，每秒 +1，`elapsedSeconds = offset + tick + 1`。这是纯计数器，不依赖系统时间。

**精度分析**：
- `Observable.interval(1s)` 在正常情况下精度很高（误差 < 10ms），因为 RxJava 使用 `ScheduledExecutorService` 调度；
- **但**如果系统休眠（Doze 模式）、CPU 被抢占、或主线程阻塞，`interval` 的发射可能会延迟或合并。此时 `tick` 的累计值会小于实际经过的时间。

**为什么没用系统时间差**：
- 系统时间差方案（`System.currentTimeMillis() - startTime`）在系统时间被用户手动修改时会产生巨大误差；
- 纯计数器在短时间间隔上更稳定，用户体验更好（秒级更新平滑）。

**改进方向**：如果要求更高精度，可以在 `interval` 的每次回调中同时用 `SystemClock.elapsedRealtime()` 做校验，当 `(elapsedRealtime - startRealtime) / 1000` 与 `tick` 偏差超过阈值时，用系统时间修正。但本项目作为专注计时器，秒级精度已足够，不需要这种复杂度。

## 三、数据持久化与一致性

### Q8. `Task` 表有一个冗余字段 `totalFocusSeconds`，用来避免 JOIN 查询。在你的场景中，什么情况下这个冗余字段会和 `focus_sessions` 表数据不一致？你做了哪些措施来防止不一致？

**答：**

**冗余字段的设计意图**：任务列表页的 `TaskCard` 需要显示每个任务的累计专注时长。如果没有 `totalFocusSeconds`，每次加载任务列表都需要 `LEFT JOIN focus_sessions GROUP BY task_id` 做聚合查询，性能开销大。

**可能不一致的场景**：
1. 手动修改数据库文件（开发/调试阶段）
2. `addTaskFocusSeconds` 执行成功但 `completeSession` 失败（上文已分析，Room 的 `andThen` 链保证了原子性）
3. 并发写入：两个 ViewModel 实例同时操作同一个 Task（本项目不会发生，因为一次只有一个 TimerViewModel 实例）

**防止不一致的措施**：
1. **原子化**：`FocusRepository.completeSession()` 使用 `andThen` 将 `updateSession` 和 `addTaskFocusSeconds` 串联，确保要么都成功、要么都失败。
2. **`WHERE is_completed = 0` 防重复**：`completeSessionByTaskAndStartTime` 的 SQL 条件确保不会重复更新已完成的 session。
3. **增量更新**：`total_focus_seconds = total_focus_seconds + ?` 是增量操作，不是覆盖，即使有 bug 导致重复执行，也会被第 2 条防线拦截。

**当前不足**：没有提供数据修复工具（比如手动触发 `totalFocusSeconds` 从 `SUM(focus_sessions.actual_duration)` 重新计算）。如果数据库文件损坏，需要手动修复 SQL。

### Q9. 统计查询用 `end_time` 而不是 `start_time` 做日期过滤。你的设计理由是什么？如果用户是一个夜猫子（23:30 开始，00:15 结束），这种分配方式合理吗？

**答：**

**设计理由**（`FocusSessionDao.java:178-181` 注释）：

"你 23:50 开始计时，00:10 结束，这次专注跨越了午夜。用 `end_time` 的话，这次专注会计入'结束那一天'。用户心理上会觉得：我'今天'完成的，就归今天。"

**合理性分析**：

对于夜猫子用户，这确实存在争议：
- 用户 23:30 开始专注，主观上觉得是"今天"的专注，但按 `end_time` 会计入明天；
- 这会导致用户在查看统计时，今天的专注时长比预期少，而明天的比预期多。

**但**我更倾向于认为这是合理的，原因：
1. 专注行为的**完成时间**比开始时间更有统计意义——用户关注的是"我什么时候完成了专注"；
2. 如果改为 `start_time`，会遇到对称问题：用户 00:05 开始专注，按 `start_time` 会计入今天，但用户可能认为这是"昨天没做完的事"；
3. 实际上早上 00:00-06:00 的专注记录很少，跨天 session 的比例极低，两种方案的实际差异很小。

**改进方向**：可以按比例分配——比如跨天 session 45 分钟，其中 30 分钟在昨天（23:30-00:00），15 分钟在今天（00:00-00:15），按比例拆分到两天。但这会增加数据库设计的复杂度（一条 session 拆成多条统计记录），在当前课程设计的场景下意义不大。

### Q10. 你的 `StatsViewModel` 用了一个 `loadingCount` 计数器来管理 6 个并行异步加载，而不是用 RxJava 的 `zip` 操作符。为什么不用 `zip`？`zip` 在你的场景下有什么问题？

**答：**

**不使用 `zip` 的原因**（`StatsViewModel.java:379-401`）：

`Observable.zip()` 的语义是"等待所有源都发射至少一个值，然后组合它们"。但它有一个致命问题：

**如果其中任何一个源发射了 `onError`，`zip` 会立即终止整个流，所有源都被取消订阅。**

这意味着：
- 如果 "今日总时长" 查询成功，但 "任务分布" 查询失败，`zip` 会取消所有订阅 → 用户看不到任何数据，只看到错误；
- 但实际上，用户应该能看到"加载成功的 5 项数据 + 1 项失败"，而不是全部空白。

**`loadingCount` 计数的优势**：
1. **容错性**：每个查询独立处理成功/失败，失败时 `checkLoadingDone()` 也照样调用，不阻塞其他查询；
2. **部分可用**：即使某一项加载失败，其他 5 项数据仍然正常显示；
3. **简单直观**：6 个计数器就够，不需要研究 `zip` / `combineLatest` / `merge` 的复杂组合语义。

**`loadingCount` 的实现细节**：
```java
// 每次加载完成（成功或失败）都调用
private void checkLoadingDone() {
    loadingCount++;
    if (loadingCount >= TOTAL_LOADS) { // 6
        isLoading.setValue(false);
        loadingCount = 0;
    }
}
```

`TOTAL_LOADS = 6` 对应：今日总时长、今日番茄钟数、今日完成任务数、图表数据、任务分布、连续天数。

## 四、Foreground Service 与生命周期

### Q11. `TimerService` 是一个 Foreground Service，背后还持有一个 `WakeLock`。请解释这两个机制各自的作用是什么？如果用户锁屏，计时器还能继续运行吗？如果系统内存紧张，你的 Service 会被 kill 吗？

**答：**

**Foreground Service**（`TimerService.java:124-127`）：
- 通过 `startForeground(NOTIFICATION_ID, notification)` 启动，通知栏常驻显示；
- 系统会给前台服务最高优先级，只有在极端内存不足时才会被 kill；
- 通知栏显示实时计时，用户可以随时暂停/停止，提供交互入口。

**WakeLock**（`TimerService.java:245-252`）：
- `PowerManager.PARTIAL_WAKE_LOCK`：保持 CPU 运行，但屏幕可以熄灭；
- 防止 CPU 进入深度休眠导致 `Observable.interval` 的定时器停止发射；
- 获取 24 小时超时，`acquire(24 * 60 * 60 * 1000)`。

**锁屏后能否继续运行**：能。Foreground Service 保证进程不被杀，WakeLock 保证 CPU 不休眠，`Observable.interval` 正常发射，计时正常更新。

**系统内存紧张时**：
- Foreground Service 比普通后台进程优先级高很多，但**不是绝对不被杀**。如果系统内存极度紧张（比如用户开了 10 个大型游戏），Android 可能 kill 所有进程；
- 此时 `onDestroy()` 中会执行最终安全网保存（`TimerService.java:381-395`），但 `onDestroy()` 不保证被调用（系统可能直接 SIGKILL）；
- 如果 `onDestroy()` 被调用了，`completeSessionByTaskAndStartTime` 会保存当前计时数据；
- 如果进程被直接 kill，`onDestroy()` 不会执行，但 `FocusSession` 表中仍有一条 `is_completed = 0` 的记录，下次启动时 `TimerViewModel.init()` 会检测到并恢复。

### Q12. 如果 App 的进程被系统杀死（比如内存不足），你的 `TimerService.onDestroy()` 里有安全保存逻辑。但 `onDestroy()` 不是一定会被调用的——如果进程被直接 kill，这个安全网是否会失效？你会如何改进？

**答：**

**是的，`onDestroy()` 不保证被调用**。如果系统发送 SIGKILL 信号直接终止进程，`onDestroy()` 不会执行，此时安全网失效。

**但项目有另一层防线**：`TimerViewModel.init()` 中会在下次启动时查询未完成的 session：

```java
// TimerViewModel.java:132-143
focusRepository.getUnfinishedSession()
    .observeOn(AndroidSchedulers.mainThread())
    .subscribe(session -> {
        if (session.getTaskId() == taskId) {
            resumeSession(session);
        }
    }, ...);
```

`getUnfinishedSession()` 查询 `end_time IS NULL AND is_completed = 0` 的记录。如果找到了，就调用 `resumeSession()` 恢复计时。

**但这不完美**：恢复时只能从 `session.startTime` 到当前时间计算已用时长，期间实际经过的时间可能包括系统 kill 和用户重新打开 App 之间的间隔（可能几分钟甚至几小时）。这会导致计时多算了。

**改进方案**：
1. **定期刷盘**：在 `Observable.interval` 中每 30 秒通过 `SharedPreferences` 或 DataStore 写入当前进度（taskId + elapsedSeconds），下次启动时读取并修正；
2. **WorkManager**：用 `OneTimeWorkRequest` 配合 `setExpedited()` 做定期心跳，WorkManager 比 Foreground Service 更稳定；
3. **使用 `onTaskRemoved`**：重写 Service 的 `onTaskRemoved()`，在用户从最近任务中划掉 App 时保存数据（这个方法比 `onDestroy()` 更可靠）。

当前版本中，`TimerService` 用的是 `START_STICKY`（`TimerService.java:109`），如果系统在有资源时自动重启 Service，`onStartCommand` 会被调用，但 Intent 为 null，无法恢复计时状态。这是一个已知限制。

## 五、UI 实现

### Q13. 你的统计图表是手写 Canvas 绘制的，而不是用 Vico。手写 Canvas 的好处是什么？如果要支持柱状图点击交互、缩放、动画，手写 Canvas 的维护成本会如何变化？

**答：**

**选择手写 Canvas 的原因**：
1. **更小的 APK 体积**：Vico 是一个功能丰富的图表库，引入它会增加约 500KB 的 APK 体积。本项目只用了柱状图和饼图，不需要 Vico 的全部功能；
2. **完全控制**：Canvas 绘制可以根据项目主题色动态调整，颜色、间距、动画都可以自定义，不依赖第三方库的 API；
3. **学习目的**：理解 Canvas 绘制原理是课程设计的一部分，Vico 是黑盒，Canvas 是白盒。

**手写 Canvas 的维护成本分析**：
- **当前状态**：两个图表文件（`DailyChart.kt` 约 200 行，`TaskDistributionChart.kt` 约 150 行），代码量可控；
- **如果加入点击交互**：需要用 `pointerInput` 检测触摸坐标，反向映射到数据索引，计算点击的是第几根柱子，复杂度中等；
- **如果加入缩放**：需要 `transformable` 修饰符 + 矩阵变换，Canvas 的坐标系统需要做 scale 和 translate，维护成本显著升高；
- **如果加入动画**：可以用 `Animatable` + `animateFloatAsState` 做柱状图的高度动画、饼图的分割动画，Compose 提供了良好的动画支持，成本可控。

**结论**：在当前需求下，手写 Canvas 是正确的选择。如果未来需要交互式图表，可以考虑引入 Vico 或 MPAndroidChart，但需要评估它们对 Compose 的支持程度（Vico 的 Compose 集成在 alpha 阶段）。

### Q14. 你的圆环 `sweepAngle` 计算是 `(elapsedSeconds % 3600) / 3600 * 360`，这意味着每 60 分钟转一圈。如果用户连续专注了 2 小时，圆环会转两圈再回到起点，视觉上用户无法区分"1 小时 5 分钟"和"5 分钟"。这是有意设计还是 bug？你打算怎么改进？

**答：**

**这是有意设计，但确实存在视觉歧义。**

**设计初衷**（`CircularTimer.kt:60-68`）：
- 60 分钟一圈，对应钟表的 60 分钟，视觉上像时钟走一圈；
- `coerceAtMost(360f)` 确保弧度不超过 360°；
- 配合中心的数字时间显示，用户可以通过数字精确知道时间。

**视觉歧义问题**：
- 用户专注了 5 分钟 → 弧线 30°（5/60 * 360）
- 用户专注了 65 分钟 → 弧线也是 30°（65%60=5, 5/60*360=30°）
- 两者视觉效果完全一样，用户只能通过中心数字区分

**改进方案**：
1. **多层环形圈**：类似 Apple Watch 的活动圆环，用多个同心圆环分别表示"分钟"和"小时"。外圈 60 秒转一圈，内圈 60 分钟转一圈；
2. **不重置弧线**：改为 `sweepAngle = (elapsedSeconds / 3600f * 360f).coerceAtMost(360f)`，让弧线在 1 小时内线性增长到满圈，满圈后保持不动。这样 65 分钟时弧线满圈，用户可以直观看到"已经超过 1 小时"；
3. **圈数指示器**：在中心文字下方增加一个小标签，显示"第 2 圈"或"1h 05m"，明确告知用户已经超过一圈。

推荐方案 2，实现改动最小且体验最好——`elapsedSeconds / 3600f * 360f`，去掉 `% 3600`。

### Q16. 如果用户快速切换任务（比如在 1 秒内连续点击"开始工作"→"停止"→"开始工作"→"停止"），你的三态机 + 数据库异步写操作会不会出现状态错乱？你如何保证 UI 状态和数据库状态的一致性？

**答：**

**状态机层面的保护**：

`toggleTimer()`（`TimerViewModel.java:157-164`）是关键防线：

```java
public void toggleTimer() {
    TimerState current = timerState.getValue();
    if (current == TimerState.IDLE || current == TimerState.STOPPED) {
        startTimer();
    } else if (current == TimerState.RUNNING) {
        stopTimer();
    }
}
```

**快速点击的时序分析**：

```
点击1: IDLE → 调用 startTimer() → timerState = RUNNING
点击2: 当前 RUNNING → 调用 stopTimer() → timerState = STOPPED
点击3: 当前 STOPPED → 调用 startTimer() → timerState = RUNNING
点击4: 当前 RUNNING → 调用 stopTimer() → timerState = STOPPED
```

由于 `timerState.setValue()` 是同步操作，每次点击时读取到的状态已经是上一次操作更新后的状态，所以状态转换是**串行**的，不会出现"同时创建两个 session"或"对一个 session 停止两次"的情况。

**数据库层面的保护**：

1. 每次 `startTimer()` 创建新的 `FocusSession`，`sessionStartTime` 是当前毫秒时间戳，即使两次点击间隔 1ms，`startTime` 也不同；
2. 安全网路径的 `WHERE is_completed = 0` 确保不会重复更新已完成的 session；
3. `currentSessionId` 在 `stopTimer()` 末尾被重置为 `-1`（`TimerViewModel.java:300`），确保下次 startTimer 时不会复用旧 ID。

**存在的风险**：如果用户在 `stopTimer()` 的异步保存操作完成前再次点击"开始"，`currentSessionId` 已经被重置为 -1，`offsetSeconds` 也在 `startTimer()` 中重新计算，逻辑上是正确的。但 `disposables` 中的旧保存操作仍在执行，可能会与新 session 的保存操作并发。不过由于每条 session 的 `startTime` 不同，`completeSessionByTaskAndStartTime` 的 WHERE 条件不会匹配到错误的 session。

## 七、综合评价

### Q17. 如果让你重新做这个项目，你会改变哪三个设计决策？为什么？

**答：**

**1. 从 RxJava 迁移到 Kotlin Coroutines + Flow**

现在回头看，RxJava 在这个项目里是"杀鸡用牛刀"。项目只用到了 `Observable.interval`、`Flowable`、`Single`、`Completable` 这些基本功能，Kotlin 协程完全可以替代：
- `Observable.interval(1s)` → `flow { while(true) { emit(Unit); delay(1000) } }`
- `Flowable<List<Task>>` → `Flow<List<Task>>`（Room 原生支持）
- `CompositeDisposable` → `viewModelScope` 自动管理生命周期
- 减少约 200 行样板代码（`subscribeOn/observeOn`、`CompositeDisposable` 管理）

**2. 去掉 Task 表的 `totalFocusSeconds` 冗余字段**

反范式设计带来的维护成本高于性能收益。在任务列表只有几十个条目的情况下，`LEFT JOIN + GROUP BY` 的查询性能完全可以接受。去掉冗余字段后，不需要维护 `addTaskFocusSeconds` 和 `completeSession` 的原子性，数据一致性风险为零。

**3. 为 TimerService 和 ViewModel 建立双向通信**

当前 ViewModel 和 Service 是"互不通信"的——ViewModel 发送 Intent 启动 Service，Service 独立运行，两者各自保存数据。这导致：
- 用户在通知栏点击暂停后，ViewModel 不知道（UI 状态不同步）；
- Service 中的 `elapsedSeconds` 和 ViewModel 中的 `elapsedSeconds` 是两个独立值。

改进方案：使用 `ServiceConnection` + `Binder` 或 `LiveData` + `BroadcastReceiver`，让 ViewModel 和 Service 保持状态同步，用户从通知栏操作后 UI 也能实时更新。

### Q18. 这个项目最有技术含量的部分是什么？你最自豪的代码是哪一段？

**答：**

**最有技术含量的部分**：计时器的**三层数据安全网**设计。

```
第一层：TimerViewModel.stopTimer() 的双路径保存
        ├─ 正常路径：通过 sessionId 精确更新
        └─ 安全网路径：通过 taskId + startTime 匹配更新

第二层：TimerService.stopTimerAndSave() 的独立保存
        └─ 不依赖 ViewModel，Service 自己保存 session

第三层：TimerService.onDestroy() 的最终兜底
        └─ 系统 kill Service 时，如果 isRunning=true 则保存
```

这三层设计确保了在各种极端场景下（快速点击、应用被杀、系统 kill 进程）用户的专注数据都不会丢失。

**最自豪的代码**：`TimerViewModel.java:242-304` 的 `stopTimer()` 方法。

这段代码体现了：
1. **竞态条件处理**：`sessionId > 0` 判断，正常路径和安全网路径的优雅切换；
2. **offsetSeconds 设计**：`actualDuration = elapsed - offsetSeconds` 精确计算本次增量；
3. **资源管理**：`timerDisposable` 单独 dispose，不影响 `disposables` 中正在执行的保存操作；
4. **注释清晰**：每个分支都有注释说明触发场景和原因，代码即文档。

这段代码只有 60 行，但包含了状态机、异步编程、竞态处理、资源管理四个维度的设计考量，体现了 "Simple but not Simplistic" 的工程哲学。
